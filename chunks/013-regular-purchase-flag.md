# 013 — Regular-purchase flag (procurement-set director bypass below $300)

> **Flips the sub-$300 default and retires `needs_director_approval` from routing.** Today, below
> $300 a PR is approved directly unless the submitter ticked `needs_director_approval` (D23) — i.e.
> director is opt-*in*. The new rule makes director the **default** below $300 and gives procurement
> an opt-*out*: a new `is_regular` flag (set by procurement on their approval form) sends a small,
> routine purchase straight to approved. `needs_director_approval` is **retired from the routing
> condition** (column kept, unused, like `skip_dept_approval`). See decision D37 (amends D30,
> supersedes the routing half of D23).

## Goal
- **≥ $300** → Director, always (unchanged, D30).
- **< $300, non-regular (default)** → Director.
- **< $300, regular** → approved directly (Procurement is final).

"Regular" = routine small spend (cleaning supplies, consumables, etc.). The classification is made by
**Procurement during their review**, via a new `is_regular` checkbox on the Procurement approval form.

## Scope (in)
- **One new `purchase_requests` field:** `is_regular` — boolean, default **false**.
- **PR Approval workflow revision (`cv237r8h7k9`, active `369076269481984`):** edit the
  director-decision condition **`bizoy1sj87j`** (after Procurement approves). Today it is an OR (basic
  engine): `needs_director_approval == true` **OR** `quoted_total_usd >= 300`. Replace the first leaf:
  - **New OR:** `quoted_total_usd >= 300` **OR** `is_regular != true`.
  - Read `is_regular` from the same node data as the amount:
    `{{$jobsMapByNodeKey.ec2h8cqal32.data.is_regular}}` (Procurement sets it on their ProcessForm, so
    the value is on the record when this condition runs — identical mechanism to `quoted_total_usd`).
  - `!= true` (notEqual), **not** `== false`, so a null/unset value is treated as not-regular →
    Director (null-safe). The `default: false` on the field is belt-and-suspenders on top.
  - Branch orientation unchanged: **true → Director path** (`eg86l2ilhmk` → Director approval),
    **false → approved** (`jy1365pvsce`).
- **Procurement approval ProcessForm:** add `is_regular` as an **editable** field (this is the only
  place it is set). ProcessForm uid is regenerated by the revision — add the field to the new one.
- **Detail popup (`2b367dbd157`):** `is_regular` **read-only** (audit visibility).

## Scope (out)
- **Dropping the `needs_director_approval` column** — kept, unused (parallels `skip_dept_approval`).
  Drop later with explicit OK. It is already **off the create form** (user rebuilt that form in MVP012),
  so no create-form change is needed.
- **Submitter setting `is_regular`** — it is a procurement-review classification, not a submitter
  field; it stays off the create form.
- **Any config-collection / per-category auto-classification** — `is_regular` is a manual judgment
  call by Procurement, same spirit as D23. Threshold stays hardcoded `300` (D30).
- The board ≥ $15k stage (D32) and the dept/custom-approver stage (D36) — untouched.

## Dependencies
- Builds on the PR Approval workflow `cv237r8h7k9` as revised through D36/MVP012 (active `369076269481984`).
- Amends D30 (director $300 floor) and retires the routing role of D23 (`needs_director_approval`).

## Acceptance
- A1 (regression, large): PR with `quoted_total_usd >= 300` → Director, regardless of `is_regular`
  (test both with regular checked and unchecked).
- A2 (new default): PR `< 300`, Procurement approves **without** checking `is_regular` → routes to
  Director.
- A3 (opt-out): PR `< 300`, Procurement **checks** `is_regular` then approves → status `approved`
  directly, no Director task.
- A4 (boundary): exactly `300.00` → Director even if `is_regular` is checked (≥ is inclusive, and the
  `< 300` half of the regular rule fails at exactly 300).
- A5 (no-quote): PR with no quote (`quoted_total_usd` computes to 0) + not regular → Director
  (default below-300 behaviour); + regular → approved.
- A6: `needs_director_approval` no longer affects routing (a sub-$300 PR with the old checkbox set —
  if any legacy record has it — routes purely on `is_regular`/amount).
- A7 (non-admin approver usability): Procurement (Pat) can edit + submit `is_regular` on the approval
  form without a `flowModels:save` 403 (the revision carries the existing comment models).
- Manual verification by user (Pat/Oliver/Dana test users).
